The end-to-end process for building AI agents that work — from defining "good" to shipping with confidence.

The Idea

Most teams build agents first and test later — if they test at all. Eval-driven development flips this: define what "good" looks like before you build, write test cases before you write the prompt, then iterate with data instead of intuition. The eval suite becomes your specification, your regression safety net, and your ship/no-ship decision gate.

What happens

  • Define the agent's purpose, users, and context
  • Inventory knowledge sources and capabilities
  • Set explicit boundaries — what it must NOT do
  • Establish measurable success criteria
  • Assess risk profile (low / medium / high)
  • Determine if behavior varies by user role

Output: Agent Vision

  • Purpose — one sentence on the core problem it solves
  • Users — who, in what context
  • Knowledge & Data — planned or actual sources
  • Core Capabilities — 3-5 things the agent should do
  • Boundaries — what it must never attempt
  • Success Criteria — these become your eval targets
Why this matters: Most teams have never written down what "good" looks like for their agent. This Agent Vision becomes the foundation for everything — the eval plan, the test cases, and eventually the agent's system prompt. Every test you write ties back to what you define here.

Zone Model for Prioritization

  • Classify scenarios into 3 zones based on Priority + Value + Effort
  • Zone 1: "This Must Work" — highest value, full investment, clear acceptance criteria. Eval every change.
  • Zone 2: "Real but Lower Priority" — scoped capabilities, intentionally lower expectations. Eval before releases.
  • Zone 3: "Exploratory" — emergent patterns from production usage. Eval periodically.
  • Drag scenarios between zones in the interactive dashboard to reprioritize

Dashboard sections

  • Scenarios & Acceptance Criteria: Each scenario gets Pass = and Fail = conditions specific enough for two reviewers to agree
  • Quality Dimensions to Test: Group scenarios by dimension (Policy Accuracy, Hallucination Prevention, Privacy Protection, etc.) with draggable chips
  • Success Metrics: Zone + Scenario + Quality Dimension + Target Pass Rate — the agreement on what "good enough" means
The Zone Model is the shared language. Business wants to ship. Dev wants quality. The Zone Model gives both a way to set expectations together — Zone 1 must work, Zone 2 is intentionally lower investment, Zone 3 is exploration. If everything is Zone 1, nothing is prioritized.

Generate & Baseline

Dashboard features

  • Zone-colored tabs: Each quality dimension tab is color-coded by dominant zone (red=Zone 1, yellow=Zone 2, blue=Zone 3)
  • Scenario groups: Test cases grouped by scenario, showing zone badge + acceptance criteria (Pass in green, Fail in red)
  • VERIFY highlights: AI-generated factual content marked in yellow — click to edit and check against real knowledge sources
  • Suggested test methods: Set at the dimension level, not per-row

What gets generated

  • Test cases organized by quality dimension with nested scenario groups
  • Expected responses with [VERIFY] markers for facts to check
  • Acceptance criteria carried forward from Stage 1 for each scenario group
  • CSV files importable into Copilot Studio evaluation
What you'd miss without this: Most teams write 5-10 happy-path tests. The generator produces 20-30 cases across adversarial attacks, hallucination traps, robustness tests, and safety boundaries. VERIFY markers flag AI-generated facts so you don't ship untested claims.

3
Run
Execute tests against the live agent and collect scores
Requires running agent

What happens

  • Import CSV test sets into Copilot Studio's Evaluation tab
  • Run evaluation against the published agent
  • Copilot Studio scores each case using the specified test method
  • Export results CSV with scores, pass/fail, and explanations
  • Alternative: run via DirectLine API with eval-runner.js for CI/CD integration

How scoring works

  • General Quality: LLM judge scores on Relevance, Groundedness, Completeness, Abstention (0.0-1.0)
  • Compare Meaning: Semantic equivalence between expected and actual (0.0-1.0)
  • Keyword Match: Code-based string matching — are required terms present?
  • Exact Match: Code-based string equality
  • Custom: Domain-specific criteria with your own labels
Run at least 3 times. AI agents are non-deterministic. A single run might mislead you. Microsoft recommends 3 runs minimum: ±5% variance is normal, ±10% needs investigation. Always export results — they're only available in Copilot Studio for 89 days.

4
Interpret & Improve
Compare results against success metrics, triage failures, decide what to fix
Requires eval results

Dashboard features

  • Success Metrics Status: Zone summary cards (e.g., Zone 1: 5/8 met) + scenario table comparing targets vs actuals
  • Expandable rows: Click any scenario to see test case details — question, expected, actual, score, pass/fail
  • Human Judgement: Agree/Disagree per test case — override the LLM judge when you know better
  • Failure Triage: Root cause classification (Eval Setup / Agent Config / Platform Limitation)
  • Top Actions & Patterns: Prioritized fixes + systemic issue detection

Human-in-the-loop

  • Each test case shows the LLM judge's Rationale explaining the score
  • Select Disagree to override — automatically moves the case to "Eval Setup — Human Disagrees" in triage
  • This prevents wasted effort fixing the agent when the eval setup was wrong
  • At least 20% of failures in a new eval are eval setup issues, not agent issues
No more SHIP/BLOCK verdicts. Instead, you see exactly which success metrics were met per zone. Zone 1: 5/8 tells you more than "ITERATE" — you know which 3 scenarios need work and what the target gap is. Deploy when Zone 1 targets are met at acceptable rates.

The Deploy Decision: Zone-Based

Instead of a single SHIP/BLOCK verdict, deployment decisions are made per zone. The question is: "At what point does this agent create more value than it costs?"

Zone 1 Met?

All "must work" scenarios meet their targets. Safety + core business passing at acceptable rates.

Deploy when Zone 1 targets are met

Zone 2 Tracked

Lower-priority gaps are documented, not blockers. Improvement signals for the next iteration.

Zone 2 gaps don't block deployment

Zone 3 Explored

Emerging patterns monitored. Promote rising use cases to higher zones over time.

Research & discovery, not operations

Architecture-Aware Eval Scoping

Different agent architectures need different eval layers. The toolkit automatically scopes what to test based on complexity — each layer inherits all tests from layers below it.

Agentic (multi-step, tool use, orchestration)

Agent calls APIs, uses connectors, chains actions, makes decisions across steps
Tool selection accuracy Action correctness Error recovery Multi-turn context retention Task completion rate

RAG / Knowledge-grounded (has knowledge sources, no tools)

Agent retrieves from documents, SharePoint, websites to ground responses
Retrieval accuracy Source attribution Hallucination prevention Completeness Grounding

Prompt-level (simple Q&A, no knowledge sources)

Agent responds from system prompt and LLM knowledge only
Response quality Tone & empathy Boundary enforcement Refusal behavior Safety & red-teaming
